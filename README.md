# TravelON Transfer-Phone Bot

Бот, який цілодобово (24/7) стежить за новими **підтвердженими** заявками на
[travelon.to](https://travelon.to) і просить агентів надіслати телефони туристів
для зв'язку з трансферменами — повторюючи ту саму логіку, що й ручний процес.

Працює як **headless-браузер (Playwright) у Docker**, розгортається на **Railway**,
код — на **GitHub**.

---

## Що він робить (один цикл)

1. Заходить у TravelON (логін + пароль із секретів Railway).
2. Відкриває сторінку заявок `/book/bundle/index`.
3. Чистить фільтр «Interval of check in» і вмикає статус **Confirmed + Confirmed Print**.
4. Читає список і залишає заявки, що: створені **сьогодні й пізніше**, у країнах
   **Греція, Болгарія, Албанія, Хорватія, Іспанія**.
5. Для кожної заявки відкриває чат і перевіряє, чи **вже** просили телефон туристів
   для трансферу. Якщо ще ні — надсилає повідомлення (відділ «Бронювання»,
   тема «Надання контактів водію автобуса», режим «To everyone»).
6. У **DRY-RUN** нічого не надсилає — лише пише в лог, кому БУ надіслав.

Захист від дублів подвійний: (1) сканування історії чату; (2) локальний список
вже опрацьованих заявок (`data/sent.json`).

---

## ⚠️ Важливо знати

- **DRY-RUN увімкнено за замовчуванням** (`DRY_RUN=true`). Спочатку переконайтесь,
  що бот знаходить правильні заявки, і лише потім перемикайте на `false`.
- Селектори сторінки зроблені «на око» з того, що я бачив. **Вхід (логін) і кілька
  селекторів майже напевно треба буде підкоригувати** під реальний DOM — див.
  розділ [«Підбір селекторів»](#підбір-селекторів). DRY-RUN захищає на час налаштування.
- Пароль ніде в коді немає і не комітиться — лише як змінна Railway.

## Локальний запуск (для перевірки/налаштування)

Потрібен Node.js 20+.

```bash
cd travelon-transfer-bot
npm install
npx playwright install chromium      # локально треба завантажити браузер

cp .env.example .env                 # і впишіть свої TRAVELON_EMAIL / TRAVELON_PASSWORD
# Для спостереження за діями поставте у .env:  HEADLESS=false

npm run once                         # один цикл (DRY-RUN) і вихід
```

Дивіться лог: скільки заявок знайдено, кому БУ надіслано. Скриншоти помилок
з'являються у `data/`.

## Розгортання на Railway

> У вас уже є акаунти GitHub і Railway.

### 1. Ініціалізувати git і запушити на GitHub

Файли репозиторію вже на місці. Залишилось створити git-репозиторій
(це треба зробити **на вашому ПК** — синхронізована папка не дає це зробити
автоматично). Найпростіше — **двічі клацнути `init-git.bat`** (або в Git Bash:
`bash init-git.sh`). Скрипт зробить перший коміт.

Далі створіть порожній репозиторій на <https://github.com/new> і виконайте:

```bash
git remote add origin https://github.com/<ваш-логін>/travelon-transfer-bot.git
git push -u origin main
```

Або вручну все одразу:

```bash
cd travelon-transfer-bot
git init
git add .
git commit -m "TravelON transfer-phone bot"
git branch -M main
git remote add origin https://github.com/<ваш-логін>/travelon-transfer-bot.git
git push -u origin main
```

(`.env`, `data/` і `node_modules/` не потраплять у репозиторій — вони в `.gitignore`.)

### 2. Створити сервіс на Railway

1. Railway → **New Project** → **Deploy from GitHub repo** → виберіть цей репозиторій.
2. Railway сам побачить `Dockerfile` і збере образ (Playwright + Chromium уже всередині).
3. Відкрийте сервіс → вкладка **Variables** і додайте змінні (мінімум):

   | Змінна | Значення |
   |---|---|
   | `TRAVELON_EMAIL` | ваш email на TravelON |
   | `TRAVELON_PASSWORD` | ваш пароль |
   | `DRY_RUN` | `true` (поки тестуєте) |
   | `CHECK_CRON` | `*/30 * * * *` (кожні 30 хв) |
   | `CRON_TZ` | `Europe/Kyiv` |

   Решта змінних мають робочі значення за замовчуванням (див. `.env.example`).
4. **Deploy**. У вкладці **Logs** дивіться результат кожного циклу.

### 3. Перемкнути на «бойовий» режим

Коли в логах DRY-RUN видно правильні заявки — змініть змінну `DRY_RUN` на `false`
і збережіть (Railway перезапустить сервіс). Тепер бот надсилає реально.

### (Опційно) Збереження стану між перезапусками

Файлова система Railway ефемерна. Щоб `data/sent.json` зберігався:
**Service → Settings → Volumes → Add Volume**, mount path `/app/data`. Це не
обов'язково — перевірка історії чату й так не дає надіслати двічі.

### (Опційно) Сповіщення в Telegram

Додайте змінні `TELEGRAM_BOT_TOKEN` (від @BotFather) і `TELEGRAM_CHAT_ID`
(від @userinfobot) — після кожного циклу бот надсилатиме короткий звіт. Дуже
зручно під час DRY-RUN.

## Змінні середовища

Повний перелік із поясненнями — у файлі [`.env.example`](./.env.example).
Найважливіші:

| Змінна | За замовч. | Призначення |
|---|---|---|
| `TRAVELON_EMAIL` / `TRAVELON_PASSWORD` | — | Дані входу (обов'язково) |
| `DRY_RUN` | `true` | `true` = лише лог; `false` = надсилати |
| `CHECK_CRON` | `*/30 * * * *` | Розклад перевірок |
| `CRON_TZ` | `Europe/Kyiv` | Часовий пояс для розкладу і «сьогодні» |
| `TARGET_COUNTRIES` | 5 країн | Які країни обробляти |
| `TARGET_STATUSES` | `Confirmed,Confirmed Print` | Які статуси |
| `ONLY_TODAY` | `true` | Лише заявки, створені сьогодні й пізніше |
| `MESSAGE_TEXT` | (текст запиту) | Що надсилати |
| `MAX_SENDS_PER_RUN` | `25` | Запобіжник від лавини надсилань |
| `HEADLESS` | `true` | `false` — бачити браузер (локально) |

## Підбір селекторів

Сторінка TravelON складна, і кілька селекторів треба перевірити на живому сайті.
Усі вони зібрані в одному місці — об'єкт `sel` у [`src/config.js`](./src/config.js).

Найшвидший спосіб знайти правильні селектори — рекордер Playwright:

```bash
npm run codegen          # відкриє https://travelon.to і вікно Inspector
```

Залогіньтесь, повторіть дії вручну (фільтр, відкриття чату, надсилання) — Playwright
покаже готові локатори. Перенесіть їх у `sel` у `src/config.js`.

Селектори, які найімовірніше потребують уваги:

- `sel.login.*` — форма входу (я її не бачив, бо ви вже були залогінені). Також
  перевірте `TRAVELON_LOGIN_URL`.
- `sel.requests.statusField` / `statusOptionByLabel` — випадайка «Status».
- `sel.requests.checkInInputs` — поля дат «Interval of check in».
- `sel.requests.resultRows` / `chatIconInRow` — рядки таблиці та іконка чату.
- `sel.chat.departmentSelect` / `subjectSelect` / `textArea` / `sendButton`.

Порада: запускайте локально з `HEADLESS=false` і `DRY_RUN=true`, дивіться скриншоти
помилок у `data/`.

## Безпека та обмеження

- Бот діє від вашого імені й надсилає повідомлення реальним агентам — тримайте
  `DRY_RUN=true`, поки не переконаєтесь у коректності.
- `MAX_SENDS_PER_RUN` обмежує кількість надсилань за цикл.
- Якщо TravelON має CAPTCHA/2FA на вході — автоматичний логін не спрацює; тоді
  перейдіть на спосіб зі збереженою сесією (можу допомогти доробити).
- Це автоматизація через інтерфейс (не офіційне API), тож зміни на сайті можуть
  вимагати оновлення селекторів. Надійніший варіант — перейти на прямі HTTP-запити,
  якщо вдасться зафіксувати ендпоінти (можу зробити окремо).

├─ src/
│  ├─ index.js        # запуск + планувальник (node-cron)

├─ Dockerfile         # образ на базі Playwright (Chromium усередині)

├─ .env.example       # приклад змінних
